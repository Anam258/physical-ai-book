# Research for Physical AI & Humanoid Robotics Handbook - Part 1 Focus

## Overview
This document consolidates research findings to inform the implementation of the Docusaurus-based handbook, specifically focusing on best practices for MDX content, code blocks, and sidebar configuration.

## Docusaurus MDX Best Practices

### MDX and Components
*   **Embedding and Importing React Components:** JSX can be written directly within Markdown files to render React components. Components defined in separate files or third-party components can be imported into MDX files. Custom components within an MDX file must be explicitly exported.
*   **MDX Component Scope:** Docusaurus manages the MDX component scope through the `@theme/MDXComponents` file, which maps HTML tag names to React component implementations.
*   **Debugging MDX:** The MDX playground is a valuable tool for debugging and validating syntax due to MDX's strict format.
*   **Markdown and JSX Interoperability:** Docusaurus v3 uses MDX v3, which is mostly CommonMark compatible but stricter. Be aware of JSX alternatives or triple backticks for code blocks where CommonMark features like indented code blocks, autolinks, and HTML syntax might behave differently.
*   **Highlight Component:** Docusaurus provides a `Highlight` component for emphasizing inline text with custom background colors.
*   **Reusable Components and Markdown Partials:** Creating reusable components and importing Markdown files as components (using `_filename` prefix for partials) promotes efficient content creation.

### Styling
*   **React Styling Approaches:** Standard React styling methods are applicable.
*   **Global Styles:** Custom CSS files (e.g., `/src/css/custom.css`) can be imported globally via `docusaurus.config.js`.
*   **Inline Styles:** Use JSX syntax for inline styling (e.g., `<span style={{backgroundColor: 'red'}}>Foo</span>`).
*   **Theming and Customization:** Extensive customization is possible by overriding styles, layouts, and components through the `/src/theme` directory and `docusaurus.config.js`.

### Technical Content Best Practices
*   **SEO Friendly:** Static HTML files are SEO-friendly, allowing page-specific optimizations.
*   **Search Functionality:** The entire Docusaurus site is searchable.
*   **Document Versioning:** Docusaurus supports document versioning for synchronizing documentation with project releases.
*   **Internationalization (i18n):** Built-in support for translating the site.
*   **Modern Tooling:** Docusaurus uses modern tooling, transforming Markdown into React components for interactive documentation.
*   **Front Matter:** Use front matter for metadata in Markdown files.

## Docusaurus Code Block Best Practices

### Language Highlighting
*   **Prism.js Integration:** Docusaurus uses Prism.js for syntax highlighting, supporting over 300 languages.
*   **Specifying Language:** Indicate the language after opening backticks (e.g., ````jsx````).
*   **Adding Languages:** Configure `additionalLanguages` in `prism` settings within `docusaurus.config.js` for non-default languages.
*   **Line Highlighting:** Use metastrings (e.g., `{1,3-5}`) or magic comments for specific line highlighting.
*   **Themes:** Change Prism theme and dark theme in `docusaurus.config.js`.

### Line Numbering
*   The `CodeBlock` component offers built-in support for displaying line numbers.

### Copy Buttons
*   Code blocks include copy buttons by default. Considerations exist for preventing copying of command-line prefixes.

### Runnable Examples
*   Docusaurus can be extended with interactive code editors and tabs for runnable examples.

## Docusaurus Sidebar Best Practices

### Defining and Generating Sidebars
*   **`sidebars.js`:** Primarily defined here, exporting a dictionary of sidebar objects.
*   **Autogenerated Sidebars:** Automatically generate based on `docs` filesystem structure using `type: 'autogenerated'` and `dirName`.

### Custom Items and Nesting
*   **Item Types:**
    *   **Doc:** Links to a documentation page.
    *   **Link:** Links to any internal or external page.
    *   **Category:** Creates a collapsible dropdown menu for nesting.
    *   **HTML:** Renders pure HTML.
    *   **Ref:** Links to a doc page without automatic navigation generation.

### Best Practices for Ordering and Customization
*   **Number Prefixes:** Prefix document/folder names with numbers (e.g., `01-Introduction.md`) for ordering.
*   **Metadata:** Use `sidebar_position` front matter in Markdown files and `_category_.json` or `_category_.yml` for folders for more robust ordering and metadata.
*   **`customProps`:** Optional object for site-specific customizations.
*   **`key` attribute:** Unique `key` for i18n translation.
*   **Custom `sidebarItemsGenerator`:** For highly specific generation logic.

### Multiple Sidebars and Top Navigation
*   Supports defining multiple sidebars for diverse documentation sections, linking top navigation items to different sidebar configurations.

### Versioning
*   Handles documentation versioning, configuring `dirName` to `versioned_docs/<version>` for autogenerated sidebars.
